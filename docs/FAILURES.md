# Amagra — Failures, Invariants & Anti-Patterns

Confirmed failure modes, architectural invariants that must never be violated, and patterns to avoid. Every entry is either observed in practice or a design invariant whose violation causes silent corruption.

---

## Part 1 — Architectural Invariants (Never Violate)

### F-01: Only `apply_learning_update()` modifies agent weights

**Rule:** No code path outside `learning.apply_learning_update()` may write to the `agent_weights` table or call `decision_weights.adjust()` / `update_ema()` directly as a production path.

**Why:** Multiple write paths produce inconsistent calibration accounting. The calibration EMA, regret tracking, instability composite, and bounded delta are all computed together inside `apply_learning_update()`. Calling `adjust()` directly bypasses calibration and decay. Weight history becomes uninterpretable.

**Exception:** `reset()` is allowed for test setup. `adjust()` is allowed for the non-chosen agent decay inside `apply_learning_update()` itself only.

---

### F-02: Never use Level 3 (similarity-based) session joins as training labels

**Rule:** Traces with `label_trustworthy=False` must not be used as supervised training targets for any learned model.

**Why:** Level 3 joins use `SequenceMatcher ≥ 0.85` similarity — probabilistic matches. Two different queries with similar text can be incorrectly linked. Training on incorrectly joined traces corrupts the learned router silently.

```python
traces = [t for t in all_traces if t.get("label_trustworthy", False)]
```

---

### F-03: Never retrain the learned router on eval/seed data

**Rule:** The 291 eval decisions in the trace dataset must not be used as training data for `learned_router.py`.

**Why:** Eval decisions are from `auto_train.py` — generated by seeding pre-defined prompts to measure routing accuracy. Training on them teaches the router to recognize eval prompts, not real user queries — inflated eval accuracy, poor generalization.

**Rule:** Train only on `real_session` traces. Wait until 100+ real sessions before the learned router is meaningful.

---

### F-04: Never bypass the memory filter gate

**Rule:** All memory saves must go through `memory_filter.py` before calling `memory_db.save()`.

**Why:** The filter gate rejects fluff, near-duplicates (similarity > 0.92), and content below minimum length. Bypassing it produces junk memories that inflate retrieval noise and corrupt quality averages.

```python
# WRONG
memory_db.save(agent="python_dev", content="ok", mem_type="code")

# CORRECT
memory_filter.maybe_save(agent="python_dev", content=response, mem_type="code")
```

---

### F-05: Never prune memories with use_count > 0

**Rule:** The pruning gate `quality < 0.55 AND use_count = 0` must not be weakened to `quality < 0.55` alone.

**Why:** A memory that has been retrieved at least once has demonstrated relevance to some query. Even if quality is low, it may provide correct associations. Deleting recalled memories based on quality alone removes evidence the system is actively using.

---

### F-06: Never disable `AGENTIC_USE_SIGNAL` on live requests

**Rule:** `AGENTIC_USE_SIGNAL=0` disables QuerySignal routing entirely — falls back to the keyword scorer. Only for ablation experiments on test data.

**Why:** Disabling signal routing drops routing accuracy from ~97% to ~85% on real queries. Every real request while the flag is set is potentially mis-routed.

**Use:** Set only before running `ablation_eval.py`. Unset immediately after.

---

### F-18: Never use CSS `!important` to override inline styles for stateful UI elements

**Rule:** Any CSS hover/focus rule that uses `!important` on `background` or `color` must not target elements whose inline style encodes runtime state (open, active, selected).

**Why:** CSS specificity means `!important` beats inline styles regardless of component state. In `MenuBar.jsx` this caused the open-menu button's selection color to be overridden by the hover rule on every hover — state visually correct before hover, broken during hover.

**Correct pattern:** Remove the CSS hover rule. Track hover state in React. Compute `background` inline with explicit priority: `open ? selectionColor : hovered ? hoverColor : "transparent"`.

---

### F-19: Layout centering must not be applied at the tab root level

**Rule:** No tab component may use `maxWidth + margin: 0 auto` at its outermost JSX element.

**Why:** Some tabs at `maxWidth: 860`, others at `maxWidth: 980` causes a jarring horizontal jump on tab switch. `App.js` is the single layout authority — `padding: 24px 28px`, `overflow: auto`, for all non-chat tabs.

**Exception:** Inner elements (chat message bubbles, article columns in ResearchTab) may use `maxWidth` for readability. Never at the root element a tab component returns.

## Part 2 — Known System Limitations

### F-10: Filesystem and file-path questions

**Fails on:** "Where is my project?", "What files did I modify recently?"

**Cause:** No tool access to the filesystem. phi4-mini generates plausible-sounding but fictional file paths.

**Workaround:** Use the terminal. Do not ask the system about filesystem state.

---

### F-11: Real-time information

**Fails on:** "What happened today?", "Is X service down?", "Latest version of Y?"

**Cause:** phi4-mini has a knowledge cutoff and no internet access.

**Workaround:** Use a web browser for current information.

---

### F-12: Cross-session memory of specific files or edits

**Fails on:** "Remember when I was working on that auth module last Tuesday?"

**Cause:** The system has semantic memory of content (what it learned), not episodic memory of your specific files. It cannot recall a file you opened unless the content was saved as a memory with sufficient quality.

**Workaround:** Paste the relevant code into the current session. Use pinned context.

---

### F-13: Multilingual input

**Fails on:** Questions written in non-English languages — specifically observed with Albanian.

**Observed:** Mixed-language response + profile leak ("user context: imagine teaching students…")

**Cause:** phi4-mini has weak multilingual instruction-following. The English system prompt overrides the input language.

**Workaround:** Write queries in English.

---

### F-14: Exact calculation tasks

**Fails on:** "What is 2^64?", "Calculate SHA256 of this string", "Days between Jan 5 and Jul 22?"

**Cause:** LLMs don't reliably compute exact arithmetic. phi4-mini will return a confident but wrong answer.

**Workaround:** Ask for the code to do the calculation rather than the result.

---

### F-15: Tasks requiring external API calls or live data

**Fails on:** "Check if my server is reachable", "Query my database", "Send an email to…"

**Cause:** No external tool access. Can write code that does these things but cannot execute network calls.

**Workaround:** Ask for the code. Execute it yourself.

---

### F-16: Very short single-word queries

**Partially fails on:** "Python?" or "Blazor?"

**Cause:** One keyword gives `conf=0.33` — routing works but `answer_shape="factual"` routes to terse, which returns a definition when you may want code or explanation.

**Workaround:** Add context. "Python?" → "How does Python handle memory management?"

---

### F-17: Compound queries with >2 domains

**Partially fails on:** "Build a Python backend, deploy on Ubuntu, expose via nginx, and add a Blazor frontend"

**Cause:** Deep Pipeline v1 fans out to a maximum of two domain agents. Third domain is silently dropped.

**Workaround:** Split into separate messages by domain, or accept that compound pipeline v1 handles pairs only.

## Part 3 — Patterns to Avoid

### P-01: Don't use the `terse` agent for anything requiring explanation

`terse` has no memory and no reflection. Its system prompt is deliberately minimal — optimised for 1–2 sentence factual lookups. "Explain neural networks" to terse returns an inadequate summary.

**Use terse for:** "What is the default Python dict ordering?" · "What port does HTTPS use?"

---

### P-02: Don't run dual-trajectory on non-code tasks

`dual_trajectory.py` gates on `is_code_task()`. Calling it directly on an explanation query doubles LLM calls with zero improvement — the critic has no meaningful basis to compare two explanations.

---

### P-03: Don't create tasks with vague prompts expecting the agent to scope them

The task queue agent receives a single prompt and cannot ask clarifying questions. Vague prompts produce broad, unfocused outputs with low-signal quality ratings.

**Rule:** A task prompt must be completable by a developer who cannot contact you.

---

### P-04: Don't force-route systematically as a substitute for fixing routing

Using force-route to work around a systematic misrouting doesn't fix the root cause — `apply_learning_update()` still uses the original (wrong) routing confidence and regret, so the brain doesn't learn.

**Rule:** If you force-route the same query type more than 3 times, add the missing keyword to `query_normalizer.py` instead.

---

### P-05: Don't run the 100-prompt eval during an active usage session

`auto_train.py --eval-only` fires 100 queries in rapid succession, dominating the calibration history and pushing all agent weights down via proxy signals.

**Rule:** Run eval only on an empty session with no real queries pending.

---

### P-06: Don't interpret post-eval weight drops as agent quality signals

After any eval run, weights drop below 1.0 for all agents — this is an artifact of proxy performance signals at scale, not evidence of poor performance. Weight recovery happens through real sessions.

**Rule:** Never make architectural decisions (removing or merging agents) based on post-eval weight values alone.
